# .asc Schematic Generation Rules

Mandatory rules for every `.asc` file generated by `scripts/gen_asc.py` or equivalent
schematic-generation logic. These rules were hardened through iterative work on the
`Optocoupler_Digital_Input.asc` schematic and reflect real failures that occurred when
they were absent.

---

## 1. Pin-to-wire clearance (stub rule)

Every component pin must have a straight stub wire of **STUB = 48 px = 3 grid units**
before any junction, bus, or connecting wire lands on it.

```
pin ──── 48 px stub ──── junction / bus / connecting wire
```

Rules:
- Stubs are strictly orthogonal — horizontal or vertical only, no diagonals.
- The stub wire must be aligned exactly on the pin's axis (no offset).
- The stub end (not the pin origin) is where junctions and connecting wires land.
- **Exception:** if placing the pin directly at a node is the only way to avoid
  routing through the component body (see Rule 2), skip the stub on that pin and
  place the pin at the node directly. Document this in a comment.

---

## 2. Component body clearance — the critical rule

**Wires must never cross, overlap, or pass through a component symbol body.**
Treat every symbol body as a forbidden routing zone.

### How to determine safe stub direction

A stub is safe only if it exits *away from* the body interior. For most primitives:

| Component | Pin | Safe stub direction |
|---|---|---|
| Resistor R0 (vertical) | A (top, local y=16) | UP (`sup`) |
| Resistor R0 (vertical) | B (bottom, local y=96) | DOWN (`sdn`) |
| Resistor R270 (horizontal) | A (left) | LEFT (`sleft`) |
| Resistor R270 (horizontal) | B (right) | RIGHT (`sright`) |
| Capacitor R0 | A (top, local y=0) | UP (`sup`) |
| Capacitor R0 | B (bottom, local y=64) | DOWN (`sdn`) |
| Diode R0 | P (top, local y=0) | UP (`sup`) |
| Diode R0 | N (bottom, local y=64) | DOWN (`sdn`) |
| Voltage source R0 | P (local y=16, "top") | UP (`sup`) |
| Voltage source R0 | N (local y=96, "bottom") | DOWN (`sdn`) |
| NPN R0 | C (local y=0, top-right) | UP (`sup`) |
| NPN R0 | B (local x=0, left) | LEFT (`sleft`) |
| NPN R0 | E (local y=96, bottom-right) | DOWN (`sdn`) |
| PMOS R0 | D (local y=0, top) | UP (`sup`) ✓ |
| PMOS R0 | S (local y=96, bottom) | DOWN (`sdn`) ✓ |
| PMOS R0 | G (local x=0, left) | LEFT (`sleft`) ✓ |
| PC817 (opto) | A, K (left side) | LEFT (`sleft`) |
| PC817 (opto) | C, E (right side) | RIGHT (`sright`) for C; DOWN (`sdn`) for E |

> **PMOS S-pin warning:** D and S share the same x. A `sup` stub from S (bottom pin)
> goes *toward* D (top pin) — directly through the body. **Always use `sdn` for PMOS S.**
> After the downward stub, detour in x before routing upward to avoid re-crossing the body.

### Body-crossing detection checklist

Before emitting any stub wire `wire(px, py, px2, py2)`, verify:
1. Does the wire travel from pin *toward* the other pin on the same device?
   → **Yes = body crossing. Choose the opposite direction.**
2. Does the wire pass through the bounding box of any *other* component?
   → **Yes = body crossing. Reroute around it.**

## 3. Symbol placement derived from pin geometry

**Always derive `SX, SY` from where the pin must land — never hardcode and hope.**

Pattern for a component whose pin P must land on node `(NX, NY)`:

```python
# pin P has local offset (LX, LY) under rotation R
# pin(SX, SY, LX, LY, R) == (NX, NY)
# → SX = NX - rot(LX, LY, R)[0]
# → SY = NY - rot(LX, LY, R)[1]
```

Common derivations:
```python
# Vertical resistor R0, pin A (16,16) must land at (NX, NY):
SX = NX - 16;  SY = NY - 16

# Vertical resistor R0, pin A at node, with STUB clearance above node:
# pin A stub ↑ STUB px, so pin A is at node_y + STUB
SX = NX - 16;  SY = (NY + STUB) - 16

# Horizontal resistor R270, pin B (16,96) must land at (NX, NY):
# rot(16,96,'R270') = (96,-16) → SX = NX-96, SY = NY+16
SX = NX - 96;  SY = NY + 16
```

**Horizontal scale factor H:** Apply only to `SX` (and to x-coordinates of horizontal
layout constants). Never apply H to y-coordinates, pin local offsets, or STUB sizes.

## 4. Direct wires over net labels

Prefer physical wire connections. Use net labels (`FLAG`) only when a direct wire would:
- require crossing another wire or component body
- span more than ~600 px without a natural routing path
- need to cross the isolation barrier (different ground domains)

### Power and GND rails

Use horizontal bus wires with a **single** identifying flag per bus:

```
[component stub] ──── [vertical route] ──── [horizontal bus] ←── FLAG "PWR5B"
```

Do **not** place individual `FLAG "PWR5B"` at every component pin. One bus wire +
one flag is sufficient and produces a cleaner schematic.

Bus placement pattern:
1. Pick a bus y-level clear of all component bodies (typically far above or below
   the main circuit area).
2. Route each component's power/gnd pin outward (via safe stub direction) then
   vertically to the bus y-level.
3. Draw one horizontal bus wire spanning all connection x-positions.
4. Place one flag anywhere on that bus wire.

### GND isolation

When two ground domains exist (e.g., GND and GND_OUT for an isolated circuit):
- Use **separate bus wires** at the same y-level but with a **gap** between them.
- Never connect the two buses. The gap in x is the isolation barrier representation.

## 5. Routing around components

When a vertical power/supply wire must pass through the x-range of a component body:

1. Route the stub out of the pin in the safe direction (per Rule 2).
2. Detour in x by at least one body width before running the long routing wire.
3. Return to the bus at the detour x — do not return to the original x until clear.

**PMOS S-pin example** (D and S share x=2304, body y=320..416):
```
S pin (2304, 416)
  ↓  sdn stub 48 px
(2304, 464)  ← below body ✓
  → wire right to x=2368 (body right edge + margin)
(2368, 464)
  ↑  wire up to PWR24B bus at y=80
(2368, 80)  ← connects to bus ✓
```

Check that the horizontal detour segment at y=464 does not cross any other
vertical wires in that x-range before committing.

## 6. When geometry prevents full STUB clearance on both pins

For a component between two nodes where:
`node_distance < body_height + 2 × STUB`

...a full stub on both pins is geometrically impossible. Resolution priority:

1. **Place the pin at the node directly** (zero stub) if that avoids a body-crossing
   wire. This is better than routing through the body.
2. Document it with a comment: `# pA sits directly at N_BASE — no stub (geometry constraint)`
3. The other pin (the one with room) still gets its full STUB clearance.

Do **not** force a stub in the body-crossing direction just to satisfy the stub rule.
Body clearance (Rule 2) always takes priority over stub clearance (Rule 1).

## 7. `.options` convergence directives

**Do not add `.options gmin=...` or `.options abstol=...` by default.**

These can break simulations that run correctly without them. In particular:
- `gmin=1e-10` (100× the default 1e-12) adds extra leakage across every PN junction.
  For circuits containing subcircuit-based models (optocouplers, integrated regulators),
  this perturbs internal node equations and can cause transient step-size failures.
- `abstol=1e-9` loosens current convergence tolerance. In some topologies this causes
  the solver to declare convergence at a wrong operating point and then fail.

**Add `.options` only if:**
- The simulation actually fails (LTspice reports "time step too small" or singular matrix), AND
- The user explicitly asks for convergence tuning.

When adding, use the minimum change that fixes the failure. Start with `reltol=0.01`
or softer source rise/fall times before touching `gmin` or `abstol`.

## 9. Label and text placement

### 9.1 Core priority rule

**Wires always have higher priority than text.**  
Text adapts to wiring — wiring never moves to accommodate text.

### 9.2 Wire clearance

Every piece of text (component label, net label, value) must be at least  
**LABEL_CLEARANCE = 16 px (1 grid unit)** away from every wire segment,  
measured as the perpendicular distance from the text origin to the nearest wire.

When `_wire_clearance(text_x, text_y)` returns < 16, shift the text in steps of 16 px
(alternating + / − in y first, then in x) until a clear position is found.

### 9.3 WINDOW override mechanism

LTspice reads `WINDOW` lines in the SYMBOL block of the `.asc` to override the  
per-instance label positions from the `.asy` defaults. Offsets are in **screen  
coordinates** (not rotated with the symbol). Format:

```
SYMBOL <name> SX SY R
WINDOW 0 <dx> <dy> <justification> <font_size>   ← InstName
WINDOW 3 <dx> <dy> <justification> <font_size>   ← Value
SYMATTR InstName …
SYMATTR Value …
```

`WINDOW` lines **must appear before** any `SYMATTR` lines in the SYMBOL block.

### 9.4 Per-component label placement table

All offsets are screen-space (dx, dy) relative to the symbol origin (SX, SY).
Font size 2 throughout. Verified clear of wires for all placements in this table.

| Component | Rotation | WINDOW 0 (InstName) | WINDOW 3 (Value) | Notes |
|---|---|---|---|---|
| `res` | R0 (vertical) | (36, 24) | (36, 64) | Right of body |
| `res` | R270 (horizontal) | (56, −52) | (56, 16) | Above / below body |
| `cap` | R0 | (36, −8) | (36, 56) | Right of body, above top |
| `diode` | R0 | (36, −8) | (36, 56) | Right of body |
| `npn` | R0 | — (use .asy default) | — | Default (56,32)/(56,68) OK |
| `pmos` | R0 | — (use .asy default) | — | Default (56,32)/(56,72) OK |
| `voltage` | R0 | — (use .asy default) | — | Default (24,16)/(24,96) OK |
| `PC817x` | R0 | — (use .asy default) | — | Default (0,−80)/(0,80) Centre OK |

**Body bounding boxes for reference (R0):**

| Symbol | x range | y range |
|---|---|---|
| `res` vertical | SX+0 … SX+32 | SY+16 … SY+96 |
| `res` horizontal (R270, screen) | SX+16 … SX+96 | SY−32 … SY |
| `cap` vertical | SX+0 … SX+32 | SY+0 … SY+64 |
| `diode` vertical | SX+0 … SX+32 | SY+0 … SY+64 |
| `npn` | SX+0 … SX+64 | SY+0 … SY+96 |
| `pmos` | SX+0 … SX+48 | SY+0 … SY+96 |
| `voltage` | SX−32 … SX+32 | SY+24 … SY+88 |
| `PC817x` | SX−96 … SX+96 | SY−64 … SY+64 |

### 9.5 Net label (FLAG) placement

Informational net labels use a **INFO_OFFSET = 32 px (2 grid units)** stub so the  
flag coordinate sits at least one grid unit above the text bottom edge:

```python
# Upward stub — text renders above flag point (font height ≈ 16 px)
# Wire at y=400, stub 32 px up, flag at y=368 → text occupies y=352..368 → gap=32 px ✓
wire(node_x, node_y, node_x, node_y - INFO_OFFSET)
flag(node_x, node_y - INFO_OFFSET, "LABEL_NAME")
```

Use `INFO_OFFSET = 32` (not 16) to guarantee the text bottom edge clears the wire  
by at least one full grid unit.

For **power/GND bus flags** (placed at bus wire ends) the flag is on the bus wire itself.  
This is the standard LTspice convention — no offset needed for bus flags.

### 9.6 Auto-adjustment algorithm

`gen_asc.py` implements `_safe_window(sx, sy, dx, dy)` which:
1. Computes world position `(sx+dx, sy+dy)`.
2. Calls `_wire_clearance(wx, wy)` → perpendicular distance to nearest wire.
3. If distance ≥ `LABEL_CLEARANCE`, returns `(dx, dy)` unchanged.
4. Otherwise tries `dy ± k×LABEL_CLEARANCE` for k = 1, 2, 3, 4, 5 (alternating signs).
5. Returns first clear position found; falls back to original if none found.

```python
LABEL_CLEARANCE = 16   # 1 grid unit

def _wire_clearance(px, py):
    for x1,y1,x2,y2 in _wires:       # _wires built by wire() calls
        if x1==x2:                    # vertical
            d = abs(px-x1) if y1<=py<=y2 else abs(px-x1)+min(abs(py-y1),abs(py-y2))
        else:                          # horizontal
            d = abs(py-y1) if x1<=px<=x2 else abs(py-y1)+min(abs(px-x1),abs(px-x2))
        if d < LABEL_CLEARANCE: return d
    return float('inf')
```

## 8. Validation checklist (run before every `.asc` handoff)

- [ ] No wire goes from a pin toward the component's own body interior
- [ ] No routing wire passes through any component symbol body
- [ ] PMOS S-pin routes downward, not upward
- [ ] All stubs are strictly horizontal or vertical
- [ ] All coordinates are multiples of 16 (grid-aligned)
- [ ] Symbol origin is derived from pin-landing math, not hardcoded guesses
- [ ] Power/GND rails use bus wires + single flags, not per-component flags
- [ ] Isolated domains (different GND symbols) have separate bus wires with a gap
- [ ] No `.options gmin` / `.options abstol` unless simulation failed and user requested it
- [ ] No embedded `.model` card for a component that exists in LTspice lib.zip

## Session learnings (Optocoupler_Digital_Input, 2026-04)

These are the specific mistakes that were found and fixed, preserved here so they are
not repeated.

### R2 body-crossing stub (fixed)
**Problem:** `SY_R2 = N_BASE_Y - STUB - RES_A[1]` placed pA at y=800. Then
`sdn(*pA_r2)` wired from y=800 downward to y=848, passing through the R2 body (y=800..880).

**Fix:** `SY_R2 = N_BASE_Y - RES_A[1]` places pA directly at N_BASE (y=848). No stub
wire emitted for pA. pB at y=928 connects directly to GND bus at y=960.

**Rule extracted:** When node_distance < body_height + 2×STUB, place the pin at the
node directly rather than trying to stub through the body.

### PMOS S-pin body-crossing stub (fixed)
**Problem:** `sup(*pS_vt2)` wired from S at y=416 upward to y=368 — through the
PMOS body (D at y=320 to S at y=416).

**Fix:** `sdn(*pS_vt2)` to y=464 (below body), then right to x=2368, then up to
PWR24B bus. x=2368 chosen to be clear of both the VT2 body (right edge ≈ x=2352)
and RLOAD (left edge x=2416).

**Rule extracted:** PMOS S always stubs DOWN (sdn). After the downward stub, detour
in x before routing upward; verify the horizontal segment does not cross any vertical
wires in that y range.

### Per-component power flags replaced by bus wires (fixed)
**Problem:** Each power/GND connection had its own `FLAG` — e.g. `flag(*sup(*pP_v5b), "PWR5B")`.
This produces many disconnected labels rather than visible bus wires.

**Fix:** Route all same-rail pins to a shared horizontal bus wire; place one flag on the bus.

**Rule extracted:** Power/GND rails use bus wires. One bus + one flag per rail, not
one flag per component pin.

### `.options gmin=1e-10 abstol=1e-9` broke simulation (fixed)
**Problem:** These options were added preemptively. The simulation ran fine without them
but failed with them. `gmin=1e-10` perturbed the PC817 internal node equations.

**Fix:** Remove the `.options` line. The simulation runs correctly without it.

**Rule extracted:** Never add `.options gmin`/`abstol` by default. Add only when
simulation actually fails and the user requests convergence tuning.
